# PDFToMarkDown 四子应用单后端重构设计

## 背景

当前仓库已经同时具备四类能力：

- 从 GitHub 下载 Markdown 并本地化资源
- 对本地 Markdown 做正则清洗
- 对本地 Markdown 做翻译
- 将本地 Markdown 导出为 HTML / PDF

但这些能力目前主要以单页 GUI 的方式混在一起，后端逻辑也集中在 `scripts/gui-server.mjs` 中。这样虽然能用，但有几个明显问题：

- 四类能力的职责边界不清晰，页面状态和接口语义容易继续堆叠
- 用户无法把某一段能力当成“独立小工具”直接使用
- 后续加入批量处理、任务管理、独立入口页时，现有单页结构会迅速变脏
- 总入口页和独立子入口页如果直接在现有结构上硬加，最终会变成重复实现

本次目标不是改技术栈，也不是改成桌面壳，而是在保持当前 Web 方案的前提下，把项目拆成同仓库下的四个子项目，并额外提供一个总壳 UI。

## 目标

- 在同一仓库内拆出四个前端子应用：
  - 下载子应用
  - 清洗子应用
  - 翻译子应用
  - 导出子应用
- 新增一个总壳子应用，用分页或导航方式统一挂载四个子应用
- 保持单后端服务，不拆成多个后端进程
- 四个子应用都可以单独访问、单独打包静态资源、单独使用
- 总壳子应用复用同一套前后端能力，不实现第二套业务逻辑
- 下载子应用支持批量输入多个 GitHub URL，并允许自定义保存目录
- 清洗、翻译、导出三个子应用都支持手动指定顶层文件夹，再扫描目录下的 Markdown 文件并多选批量处理
- 后端任务模型统一，避免只有翻译具备任务状态、其他流程仍是同步直跑

## 非目标

- 不改为桌面应用或引入 Electron / Tauri
- 不更换当前核心运行时，继续沿用现有 Node 脚本能力
- 不在本轮引入多服务部署、微服务通信或复杂微前端方案
- 不重写核心 Markdown 处理逻辑，只做模块化重组和入口重构
- 不在本轮增加自动串联流水线，例如“下载后自动清洗再自动翻译”

## 约束与决策

### 1. 运行形态

继续使用 Web 应用方案。这是为了最大化复用现有 `gui/`、`scripts/gui-server.mjs` 和四个处理脚本，避免把重构成本浪费在运行时迁移上。

### 2. 后端形态

采用“单后端 + 多前端入口”。

这意味着：

- 后端只保留一个服务进程
- 后端同时暴露下载、清洗、翻译、导出四组接口
- 四个独立子页面与总壳页面都连接同一个后端
- 总壳只是导航和组合层，不额外复制业务实现

### 3. 批量处理模型

批量任务统一抽象为“一个批量任务下包含多个子项任务”。

- 下载页：一个 URL 对应一个任务子项
- 清洗页：一个 Markdown 文件对应一个任务子项
- 翻译页：一个 Markdown 文件对应一个任务子项
- 导出页：一个 Markdown 文件对应一个任务子项

### 4. 输入方式

- 下载子应用：输入多条 GitHub Markdown URL，并允许自定义保存根目录
- 清洗子应用：输入顶层文件夹路径，扫描后多选 Markdown 文件批量清洗
- 翻译子应用：输入顶层文件夹路径，扫描后多选 Markdown 文件批量翻译
- 导出子应用：输入顶层文件夹路径，扫描后多选 Markdown 文件批量导出 HTML / PDF

这里不把“本地三页直接选文件”作为主流程，而是明确改成“顶层文件夹扫描 -> 多选 -> 批量执行”。

## 方案对比

### 方案 A：最小改动型

保留当前 `scripts/*.mjs + scripts/gui-server.mjs + gui/` 结构，只是把单页拆成多个页面。

优点：

- 改动量最小
- 最快能看到页面拆分结果

缺点：

- 后端仍会继续堆在一个服务文件里
- 子项目边界不硬，后续维护仍然容易回到单页泥团

### 方案 B：模块化单后端型

保留一个 Node 后端服务，但明确拆分后端路由、服务层、共享模块和前端多入口。

优点：

- 改动仍控制在当前技术栈内
- 子应用独立性和共享能力能同时成立
- 适合后续继续扩批量能力、任务历史、失败重试

缺点：

- 比单纯分页拆分多一轮结构重组

### 方案 C：伪微前端型

前端和后端都按子应用彻底分目录装配，总壳再聚合。

优点：

- 隔离最强

缺点：

- 结构重复明显
- 对当前项目规模来说过度设计

### 结论

采用方案 B，即“模块化单后端型”。

这是当前需求下最平衡的选择：既满足四个子应用可独立运行，又避免为了形式上的独立把共享逻辑复制四遍。

## 目标架构

### 总体结构

- 一个后端进程
- 五个前端入口
- 四类核心处理能力继续通过现有 CLI / 脚本执行
- 所有长耗时能力统一纳入任务模型

### 前端入口

- 下载子应用
- 清洗子应用
- 翻译子应用
- 导出子应用
- 总壳子应用

其中前四个入口页可以单独打开。总壳子应用提供统一导航，让用户在一个页面内切换四个功能分页。

### 后端职责

后端只做三件事：

- 处理 HTTP 请求与响应
- 调用对应服务层能力
- 维护任务、日志、文件扫描和公共配置

后端不再继续把所有业务细节直接写在一个入口文件里。

## 目录结构设计

建议重构为如下结构：

```text
/apps
  /download-ui
  /clean-ui
  /translate-ui
  /export-ui
  /shell-ui

/server
  /routes
    download.mjs
    clean.mjs
    translate.mjs
    export.mjs
    files.mjs
    config.mjs
    jobs.mjs
  /services
    download-service.mjs
    clean-service.mjs
    translate-service.mjs
    export-service.mjs
  /shared
    paths.mjs
    response.mjs
    jobs.mjs
    validation.mjs
    files.mjs
  server.mjs

/tools
  download-markdown.mjs
  clean-markdown.mjs
  translate-markdown.mjs
  export-pdf.mjs

/config
/docs
/tests
```

### 目录边界说明

`apps/*`

- 每个目录对应一个独立前端子项目入口
- `shell-ui` 只负责导航和组合
- 子应用之间不互相复制业务逻辑

`server/routes`

- 只处理路由、参数读取、响应格式
- 不直接包含复杂业务逻辑

`server/services`

- 负责封装四类处理能力
- 负责把 CLI / 脚本执行包装成后端服务能力

`server/shared`

- 放置路径校验、任务状态管理、公共响应、文件扫描、参数校验等通用逻辑

`tools`

- 放真正可独立命令行运行的核心脚本
- 保留独立 CLI 可执行能力
- 后端服务层通过这些脚本或其导出的能力复用处理流程

## 五个页面的职责设计

### 1. 下载子应用

职责只保留一件事：批量从 GitHub 下载 Markdown，并把资源固化到用户指定的保存目录。

输入：

- 多行 GitHub Markdown URL
- 保存根目录
- 语言或其他与抓取有关的可选参数

输出：

- 每个 URL 对应一个任务子项
- 每个子项显示：
  - 原始 URL
  - 保存目录
  - 生成的本地 Markdown 路径
  - 图片抓取数量
  - 成功 / 失败状态
  - 错误信息

### 2. 清洗子应用

职责只处理本地 Markdown 清洗。

输入流程：

1. 手动填写顶层文件夹路径
2. 扫描该目录下的 Markdown 文件
3. 多选要处理的文件
4. 配置或编辑清洗规则
5. 提交批量清洗任务

输出：

- 每个文件单独生成 `*.cleaned.md`
- 每个任务子项显示匹配数量、输出路径、状态、错误信息

规则管理归属于这个页面，不应继续和下载、翻译、导出混在一起。

### 3. 翻译子应用

职责只处理本地 Markdown 翻译。

输入流程：

1. 手动填写顶层文件夹路径
2. 扫描 Markdown 文件
3. 多选要翻译的文件
4. 设置目标语言、模型、并发、批次大小、缓存文件、API Key
5. 提交批量翻译任务

输出：

- 每个文件是一个任务子项
- 文件级显示状态、输出路径、错误信息
- 文件内部继续保留块级进度信息

也就是页面既要有“总批量进度”，也要保留“单文件翻译进度”。

### 4. 导出子应用

职责只处理本地 Markdown 到 HTML / PDF 的导出。

输入流程：

1. 手动填写顶层文件夹路径
2. 扫描 Markdown 文件
3. 多选要导出的文件
4. 设置导出参数
5. 提交批量导出任务

输出：

- 每个文件对应一个任务子项
- 返回 HTML 路径、PDF 路径、图片统计、状态、错误信息

### 5. 总壳子应用

总壳子应用只负责：

- 显示统一导航
- 切换四个子应用分页
- 复用四个子应用已有的前端模块

总壳不负责复制四份表单，也不负责二次包装后端接口。

## 后端接口设计

### 下载

- `POST /api/download/scan-preview`
  - 可选预校验 URL 列表与保存目录
- `POST /api/download/jobs`
  - 提交批量下载任务

### 文件扫描

- `POST /api/files/scan-markdown`
  - 输入顶层文件夹路径
  - 返回该目录下扫描到的 Markdown 文件列表

这个接口由清洗、翻译、导出三个子应用共用。

### 清洗

- `GET /api/clean/rules`
  - 读取清洗规则
- `POST /api/clean/rules`
  - 保存清洗规则
- `POST /api/clean/jobs`
  - 提交批量清洗任务

### 翻译

- `POST /api/translate/jobs`
  - 提交批量翻译任务

### 导出

- `POST /api/export/jobs`
  - 提交批量导出任务

### 通用任务

- `GET /api/jobs/:jobId`
  - 查询一个批量任务状态
- `GET /api/jobs/:jobId/items`
  - 查询该批量任务下的所有子项
- `GET /api/jobs/:jobId/logs`
  - 查看任务日志
- `POST /api/jobs/:jobId/cancel`
  - 取消未完成任务

## 任务模型设计

### 批量任务

批量任务表示一次页面提交，例如：

- “下载这 8 个 GitHub URL”
- “清洗这个目录里选中的 12 个 Markdown 文件”
- “翻译这 5 个 Markdown 文件”

字段建议至少包括：

- `jobId`
- `jobType`
- `status`
- `createdAt`
- `startedAt`
- `finishedAt`
- `inputSummary`
- `totalItems`
- `completedItems`
- `failedItems`
- `cancelledItems`

### 任务子项

任务子项表示批量任务中的单个处理对象。

字段建议至少包括：

- `itemId`
- `jobId`
- `status`
- `source`
- `output`
- `stats`
- `error`
- `logs`

### 状态集合

统一使用以下状态：

- `queued`
- `running`
- `completed`
- `failed`
- `cancelled`

翻译当前已有的 job 机制应保留，但要被提升为统一任务模型的一部分，而不是继续作为特例存在。

## 前端交互原则

### 独立页与总壳必须同源复用

独立子应用页和总壳分页应加载同一套业务前端模块，而不是分别维护两份页面逻辑。否则后续修某个页面时，总壳和独立页会很快出现行为偏差。

### 页面只暴露本领域参数

拆分后的子应用不应再出现“一个表单承载四类流程”的情况。每页只显示与本页能力相关的参数和结果。

### 批量任务可观察

所有批量页都要具备：

- 当前任务状态
- 子项列表
- 成功 / 失败统计
- 结果链接
- 错误信息

翻译页额外保留块级进度展示。

## 关键实现顺序

### 第一阶段：先拆后端

把当前 `scripts/gui-server.mjs` 中的以下内容拆开：

- 路由
- 文件扫描
- 规则读写
- 任务管理
- 脚本调用
- 路径安全校验

先建立稳定的服务边界，再动前端。

### 第二阶段：抽前端共享能力

从当前 GUI 中抽出公共能力：

- API 请求封装
- 任务轮询
- 错误处理
- 状态展示
- 结果链接展示
- Markdown 扫描结果列表

### 第三阶段：分别做四个子应用

按下载、清洗、翻译、导出四页独立实现入口，不再共用一个大表单。

### 第四阶段：再做总壳

总壳只作为组合层，在四页稳定后再接入，避免总壳反过来拖累核心页面重构。

### 第五阶段：补测试与回归

重点验证：

- 顶层文件夹扫描是否准确
- 多文件批量处理是否相互隔离
- 翻译任务轮询是否稳定
- 下载自定义保存目录是否安全
- 导出 HTML / PDF 是否保持现有产物质量

## 风险与取舍

### 1. 当前后端耦合过高

`scripts/gui-server.mjs` 现在同时承担路由、业务、任务、文件系统和页面分发职责，是本次重构的最大结构风险。如果不先拆后端，前端拆成四页后仍会被同一个脏入口反噬。

### 2. 纯 Web 方案下的目录选择能力有限

当前是 Web 方案，不是桌面壳。浏览器无法像桌面程序那样稳定获得本地任意目录选择能力，因此本轮主流程应明确采用“手动填写顶层文件夹路径并扫描”的方案，而不是把系统目录选择器当成必要前提。

### 3. 任务模型必须统一

如果继续保留“翻译是异步任务、下载清洗导出是同步请求”的混合模式，四个子应用的用户体验会不一致，代码也难复用。因此本轮应接受一定的改造成本，把三条同步流程也统一纳入任务模型。

### 4. 总壳不能变成第二套实现

如果总壳只是重新拼装一份四页表单，后续维护成本会翻倍。因此必须坚持“总壳只组合、子应用负责业务”的边界。

## 测试策略

### 后端

- 文件扫描接口测试
- 清洗规则读写测试
- 路径安全校验测试
- 各类任务创建与状态推进测试
- 下载 / 清洗 / 翻译 / 导出服务层测试

### 前端

- 四个子应用入口加载测试
- 各页面表单参数提交测试
- 批量任务状态展示测试
- 总壳导航与分页切换测试

### 运行时验证

- 下载页：多 URL + 自定义目录真实跑通
- 清洗页：顶层目录扫描 + 多文件清洗跑通
- 翻译页：顶层目录扫描 + 多文件翻译 + 进度轮询跑通
- 导出页：顶层目录扫描 + 多文件导出 HTML / PDF 跑通
- 总壳页：四个分页都能正常访问和提交任务

## 最终结论

本项目应重构为“单后端服务 + 四个独立子应用 + 一个总壳 UI”的同仓结构。

推荐的核心原则是：

- 不改 Web 技术路线
- 不拆多后端服务
- 先拆后端，再拆前端
- 四个子应用各自只负责一个领域
- 下载页支持多 URL 和自定义保存目录
- 清洗、翻译、导出统一改成“顶层文件夹扫描 -> 多选 -> 批量执行”
- 所有长耗时流程统一收敛到一个任务模型
- 总壳只做组合层，不复制业务实现

这套设计能在不大幅推翻现有项目的前提下，把当前单页工具稳定演进为可独立使用、也可统一组合的多入口系统。
